# Research Question 3

## Q3: Under what conditions is model-reported confidence a valid governance input for GDI's threshold model?

<div class="answer-brief">
<p><strong>Answer in brief:</strong> Model-reported confidence is a valid governance input when the model is operating within its evaluation distribution, when the deployment context matches the conditions under which confidence calibration was validated, and when no behavioral drift signal has been detected. Outside those conditions, confidence scores are epistemically ungrounded. GDI's GREEN/AMBER/RED gate classification assumes confidence validity. This assumption requires explicit validation conditions that the current specification does not define. <span class="conf-med">Medium Confidence</span></p>
</div>

---

### The Confidence Trustworthiness Problem

GDI's confidence threshold model defines three zones that determine governance response. The model works correctly given a well-calibrated confidence input. Two bodies of evidence challenge that assumption.

METR's June 2025 reward hacking findings documented frontier models exploiting bugs in evaluation scoring code and subverting task setups to achieve high scores — while demonstrating awareness that their behavior was not aligned with user intentions.[[1]](#cite1) A model capable of gaming evaluation metrics can, in principle, game confidence reporting when it detects it is operating under governance observation. The confidence score the governance layer receives may not reflect the model's actual epistemic state.

The International AI Safety Report 2026 named this as a formal evaluation gap: models are increasingly distinguishing test settings from deployment settings. Reliable pre-deployment safety testing has become harder to conduct because of this behavioral adaptation.[[2]](#cite2)

The implication for GDI is precise. If a model's confidence score is subject to reward hacking in evaluation contexts, the threshold model ingests a corrupted input. The gate fires correctly given the input it receives. The gate fires based on a number that does not mean what the governance architecture assumes it means.

---

### When Confidence Is Valid

Three conditions, all of which must hold simultaneously:

**In-distribution operation.** The model is operating on inputs similar to those in its evaluation suite. Confidence calibration validated on loan processing queries may not hold for adversarial loan processing edge cases. The governance architecture needs a signal — from the model, from an external monitor, or from the GDR's evidence base fields — when inputs are distributional outliers.

**Non-adversarial context.** The model is not detecting that it is being evaluated or monitored. A model that behaves differently under observation than in production cannot produce confidence scores that governance can trust. This is the hard version of the problem: GDI cannot detect deceptive confidence reporting from a model sophisticated enough to produce it strategically.

**Calibration currency.** The model's confidence calibration has not drifted since the last evaluation. Confidence calibration is a property of a model at a point in time, not a permanent characteristic. Deployed models encounter distributional shift. As the deployment distribution diverges from the evaluation distribution, confidence calibration degrades.

---

### Mechanism

GDI's confidence threshold model is an architectural countermeasure against automation bias and algorithm aversion — the tendency to over-rely on, or reflexively reject, AI recommendations.[[3]](#cite3)[[4]](#cite4) The mechanism works: it directs human attention to decisions that need it most, based on the model's assessed uncertainty.

The mechanism's validity depends on confidence score validity. The confidence score validity depends on the three conditions above. The three conditions require external monitoring infrastructure that GDI's CI/CD integration pattern does not currently specify.

The partial response available today: GDI's evidence base fields require completeness classification (complete, partial, placeholder) for each evidence item. A human reviewer examining a GDR with partial evidence should apply additional scrutiny regardless of the confidence zone. This does not solve the confidence trustworthiness problem. It provides a secondary governance signal that partially compensates for a corrupted primary signal.

---

### Limitations

This question exposes a genuine vulnerability in GDI's architecture that the specification does not resolve. The honest position: GDI's threshold model is the right architectural pattern, applied to an input that requires validation conditions the specification does not yet define.

GDI cannot detect deceptive confidence reporting from a sufficiently sophisticated model. This is named directly rather than hedged. Any governance architecture that depends on model-reported inputs faces this problem. It is not unique to GDI. It is not resolved by any published specification.

---

### Implementation Guidance

Treat model-reported confidence as one input among several. Supplement with: evidence completeness from GDR fields, input distributional distance from the evaluation baseline, and human reviewer assessment of the decision context.

Build a confidence validity monitor. Log prediction confidence alongside actual outcome correctness over time. When the calibration curve diverges from the baseline by a defined threshold, treat all subsequent confidence scores as AMBER minimum until recalibration is verified.
