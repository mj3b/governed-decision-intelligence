# Research Question 5

## Q5: What is the mechanism by which expired evidence should propagate through a governed decision record, and how does this differ from existing data lineage approaches?

<div class="answer-brief">
<p><strong>Answer in brief:</strong> Evidence expiration is a decision-level governance trigger that existing data lineage approaches do not address. Data lineage tracks the provenance and freshness of data. Evidence expiration governs the downstream invalidation of a prior accountability record when its evidence base lapses. No published specification formalizes this mechanism. GDI's evidence completeness fields and downstream propagation map are the architectural foundation. The mechanism requires three elements: an expiration timestamp on each evidence item in the GDR, a propagation rule defining what triggers re-evaluation, and a governance event record when expiration fires. <span class="conf-high">High Confidence</span> <span class="orig">Original Contribution</span></p>
</div>

---

### The Gap in Data Governance

Data governance addresses data quality, lineage, and freshness. A data lineage system records where data came from, what transformations it underwent, and when it was last updated. Data freshness policies define how stale data may be before it is flagged or rejected.

None of these mechanisms addresses the decision-level consequence of evidence expiration. The question data lineage answers: *is this data current?* The question evidence expiration governance must answer: *given that the evidence underlying this decision has expired, what happens to the governed record of that decision, and who is responsible for reviewing it?*

These are different questions. Data freshness is an input quality property. Evidence expiration is a governance event that affects prior accountability records.

---

### Three Scenarios Where Evidence Expiration Matters

**Regulatory precedent superseded.** A pharmaceutical company made an IND submission decision based on a specific FDA precedent for hepatotoxicity risk. That precedent was revised in a subsequent FDA guidance update. The decision was made on evidence that is now superseded. The governed record of that decision — the GDR — recorded the evidence as complete at the time. The superseding guidance creates a governance event: the evidence basis is no longer valid, the decision may need re-evaluation, and a named human must acknowledge the changed evidence landscape.

**Clinical data updated.** A clinical trial decision was made based on 90-day safety data. The 180-day data became available and showed a signal not present in the 90-day data. The original GDR recorded the 90-day data as complete. The 180-day data expires the completeness classification of that evidence item.

**Market condition shift.** A credit decision was made using a risk model calibrated on pre-2020 economic conditions. Those conditions no longer hold. The model's calibration has expired relative to the current deployment context.

In each case, existing data governance systems would flag the data as stale. None would propagate that staleness to the accountability record that relied on it, trigger a governance review, or assign a named human to acknowledge the changed evidence landscape.

---

### The Mechanism

A complete evidence expiration governance mechanism requires four elements:

**Expiration timestamp.** Each evidence item in the GDR's evidence base carries an expiration timestamp, either absolute (this guidance document expires when superseded) or relative (this data is valid for 180 days from collection date).

**Propagation rule.** When an evidence item's expiration timestamp is reached, a governance event fires. The GDR is flagged as containing expired evidence. The downstream propagation map identifies what decisions depend on this GDR. Those downstream GDRs are also flagged.

**Re-evaluation trigger.** The governance event creates a review obligation. A named human — the accountability chain owner in the original GDR — must review the decision in light of the expired evidence and either reaffirm the original decision with updated evidence or initiate a No-Go or Escalate outcome.

**Governance event record.** The review obligation, its outcome, and the updated evidence basis are recorded as a new GDR linked to the original. The original GDR is not modified — it remains the contemporaneous record of the decision as made. The new GDR is the governance record of the re-evaluation.

---

### How This Differs from Data Lineage

| Dimension | Data Lineage | Evidence Expiration Governance |
|-----------|-------------|-------------------------------|
| Object governed | Data asset | Decision accountability record |
| Trigger event | Data staleness | Evidence validity boundary reached |
| Response | Flag data as stale | Create governance review obligation |
| Named accountable party | Data owner | Decision accountability chain owner |
| Output artifact | Updated lineage record | New GDR linked to original |
| Retrospective effect | Updates current view of data | Creates obligation against prior decision record |

---

### GDI's Current State

GDI's GDR schema includes evidence completeness classification (complete, partial, placeholder) and a downstream propagation map.[[1]](#cite1) These are the foundational elements for evidence expiration governance. The current specification does not define the expiration timestamp field, the propagation rule, or the governance event record.

This is a specification gap, not an architectural gap. The architectural foundation exists. The mechanism definition is the contribution this question proposes.

---

### Limitations

This is an original contribution — no published specification defines this mechanism. The mechanism proposed here is the author's own formulation, subject to challenge and refinement. The claim "no published specification addresses evidence expiration as a decision-level governance trigger" was verified by search in May 2026 and is accurate at that date.
