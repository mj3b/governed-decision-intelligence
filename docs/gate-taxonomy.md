# GDI Gate Taxonomy

Gate classification answers a question distinct from policy evaluation: given that an action is permitted, what level of human deliberation does this specific decision require?

This question has no answer in current AI governance infrastructure. Policy engines answer "is this permitted?" Audit logs answer "what happened?" The gate taxonomy answers "what institutional accountability applied before execution?" — a question that matters most when a permitted action later produces an adverse outcome, and an auditor needs to reconstruct whether the decision that authorized it received appropriate human oversight.

The taxonomy is informed by phase-gate decision methodology from regulated industries, where decisions are discrete documented events with defined governance consequences, and by aviation's Crew Resource Management framework, which treats the decision to halt as carrying the same procedural weight as the decision to proceed.

---

## The Five Decision Outcomes

| Outcome | Definition | Governance Implication |
|---------|-----------|------------------------|
| Go | Decision proceeds with full confidence and complete evidence | Standard GDR logged. Downstream propagation tracked. |
| Conditional Go | Decision proceeds with explicit conditions, named owners, and verification checkpoints | Conditions schema-validated. Each condition carries a named owner and a deadline. |
| Defer | Decision paused due to insufficient evidence or unresolved dependencies | Re-entry criteria defined. Evidence gaps documented. Timeline required. |
| No-Go | Decision explicitly rejected with documented rationale | Re-entry logic captured. Stopping treated as risk reduction. |
| Escalate | Decision exceeds current authority scope or policy boundaries | Escalation path, receiving authority, and outcome all logged. |

Defer and No-Go are first-class outcomes with the same governance rigor as Go decisions. The decision to stop is a governed artifact, not an absence of decision. An ungoverned No-Go is indistinguishable from an ungoverned failure. The GDR makes them distinguishable.

## The Four Gate Levels

Gate levels describe what level of human deliberation a decision requires. They are determined before execution, not after.

| Gate | Name | Human Deliberation Required | Trigger Conditions |
|------|------|-----------------------------|--------------------|
| Gate 1 | Routine Execution | None at decision time. Automated documentation. | Policy allowed. Confidence in GREEN zone. Routine tool class. |
| Gate 2 | Documented Delegation | None at decision time. Upstream delegation invoked and recorded. | Policy allowed. Delegation reference present. Decision within pre-authorized scope. |
| Gate 3 | Elevated Review | Mandatory. A designated reviewer must acknowledge the artifact before or immediately after execution. | Policy allowed but: confidence in AMBER zone, elevated tool class, novel conditions, or approaches policy boundary. |
| Gate 4 | Hard Escalation | Execution halts. Human authority required before any action proceeds. | Policy denied. Confidence in RED zone. Out-of-bounds tool class. Conditions the system cannot classify. |

---

## Gate Classification Is Normative, Not Observational

Gate classification must be defined by the institution and applied by the governance layer before execution. It cannot be generated by the model being governed. Self-certification is not governance.

This constraint has a precise technical implication: the gate classifier is institutional infrastructure, not a component of the AI system it governs. An AI system that classifies its own governance level provides no accountability guarantee — the classification is as reliable as the model, subject to the same failure modes, and cannot serve as independent evidence of oversight.

The institution defines what "elevated review" means for their domain, which tools trigger Gate 4, and how confidence thresholds map to gate levels. GDI provides the architecture and enforces that thresholds are applied. Institutions provide the calibration.

---

## Relationship to Confidence Zones

Confidence zone assessment is one input to gate classification. Gate level may also be driven by:

- Tool class (write, delete, and financial-record tools may always require Gate 3 or Gate 4 regardless of confidence score)
- Policy decision (a policy denial always produces Gate 4)
- Institutional override (custom rules for specific agent types, contexts, or regulatory domains)

A GREEN zone confidence score does not guarantee Gate 1 or Gate 2. A Gate 4 designation cannot be overridden by a GREEN zone score. The confidence model informs; the gate taxonomy governs.

## GDR Field

Gate classification is captured in the `gate_classification` object of every GDR:

```json
{
  "gate_classification": {
    "gate": "gate_3_elevated_review",
    "gate_rationale": "Tool class database_write requires elevated review per institutional policy. Confidence score 0.85 is above threshold but tool-class override applies.",
    "escalation_required": false,
    "escalation_trigger": null,
    "delegation_reference": null
  }
}
```
